# TrustAccept Source Notes

These source notes informed the strategy. Verify again before using exact claims in public marketing.

## NIST CSF 2.0

NIST CSF 2.0 organizes cybersecurity outcomes using six functions:
- Govern
- Identify
- Protect
- Detect
- Respond
- Recover

Use TrustAccept language around risk governance, decision accountability, and evidence.

Source:
https://nvlpubs.nist.gov/nistpubs/CSWP/NIST.CSWP.29.pdf

## NIST SSDF

NIST SP 800-218 is the Secure Software Development Framework, focused on practices for reducing software vulnerabilities.

Use TrustAccept language around secure release risk records and vulnerability exception handling.

Source:
https://csrc.nist.gov/pubs/sp/800/218/final

## CISA KEV

CISA Known Exploited Vulnerabilities catalog is an input for vulnerability management prioritization. Use “CISA KEV-aware,” not “CISA approved.”

Source:
https://www.cisa.gov/known-exploited-vulnerabilities-catalog

## OWASP LLM / Agentic AI

OWASP LLM and Agentic AI guidance supports the need for guardrails and risk review around autonomous actions.

Sources:
https://owasp.org/www-project-top-10-for-large-language-model-applications/
https://genai.owasp.org/resource/agentic-ai-threats-and-mitigations/

## Auth0 for AI Agents

Auth0 supports AI-agent identity, Token Vault, and asynchronous authorization / human-in-the-loop approvals. SequenceNow should remain the delivery/identity rail; TrustAccept should be the decision/evidence layer.

Sources:
https://auth0.com/ai
https://auth0.com/ai/docs/intro/asynchronous-authorization
https://auth0.com/ai/docs/intro/token-vault

## Okta for AI Agents

Okta is moving toward AI-agent identity governance. This validates the agent identity/security market.

Source:
https://www.okta.com/products/govern-ai-agent-identity/

## Compliance Automation Competitors

Vanta, Drata, Secureframe, and Sprinto are broad trust/compliance/GRC platforms. TrustAccept should avoid directly competing as a full compliance automation product.

Sources:
https://www.vanta.com/trust-management-platform
https://drata.com/
https://secureframe.com/
https://sprinto.com/

## Security Automation Competitors

Tines, Torq, and BlinkOps validate security workflow automation but are broader than TrustAccept should be at launch.

Sources:
https://www.tines.com/
https://torq.io/
https://www.blinkops.com/workflow-studio

## Vulnerability Exceptions

Rapid7 and ServiceNow show vulnerability exceptions and exception workflows are already recognized categories. TrustAccept can productize the decision/evidence layer without competing directly with vulnerability management platforms.

Sources:
https://docs.rapid7.com/insightvm/working-with-vulnerability-exceptions/
https://www.servicenow.com/docs/r/yokohama/security-management/vulnerability-response/vr-exception-management.html

## Privileged Access / JIT Access

StrongDM and Apono validate approval workflows and just-in-time access. TrustAccept should not be PAM; it should document and route high-risk cyber decisions that may include access events.

Sources:
https://docs.strongdm.com/admin/access/approval-workflows
https://www.apono.io/
